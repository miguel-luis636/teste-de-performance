# 📄 PLANO DE TESTE DE PERFORMANCE

## 1️⃣ Objetivo do Teste

Descrever claramente o propósito do teste.

**Deve responder:**

* Por que o teste será executado?
* Qual risco estamos mitigando?
* Qual cenário de negócio estamos validando?

**Exemplo:**

> Validar se o sistema suporta até 2.000 usuários simultâneos mantendo p95 abaixo de 500ms e taxa de erro inferior a 1%, garantindo estabilidade antes da entrada em produção.

---

## 2️⃣ Escopo

Definir o que será e o que não será testado.

### Incluído:

* Página de Login
* API de Autenticação
* API de Checkout
* Banco de dados principal

### Não incluído:

* Integrações externas (gateway de pagamento)
* Serviços de terceiros

⚠ O escopo deve evitar ambiguidades.

---

## 3️⃣ Requisitos do Teste

Pré-condições necessárias para execução.

* Ambiente configurado e estável
* Infraestrutura documentada (CPU, RAM, instâncias)
* Banco com volume realista de dados
* Massa de dados preparada
* Monitoramento ativo
* Permissões de acesso às ferramentas

---

## 4️⃣ Tipos de Teste de Performance

Definir quais serão aplicados:

* Teste de Carga
* Teste de Estresse
* Teste de Spike
* Teste de Soak
* Teste de Escalabilidade
* Teste de Volume

Descrever para cada tipo:

* Quantidade de usuários
* Ramp-up
* Duração
* Objetivo específico

Exemplo:

**Teste de Carga**

* 1.500 usuários simultâneos
* Ramp-up de 10 minutos
* Duração: 30 minutos

---

## 5️⃣ Métricas de Performance

Definir indicadores que serão analisados.

### Aplicação:

* Tempo de resposta médio
* Percentis (p95, p99)
* Throughput (RPS)
* Taxa de erro (%)
* Tempo de processamento

### Infraestrutura:

* Uso de CPU
* Uso de memória
* I/O de disco
* Conexões de banco
* Latência de rede

⚠ Métricas devem ser definidas antes da execução.

---

## 6️⃣ Plano de Execução

Descrever como o teste será conduzido.

1. Validar ambiente
2. Iniciar monitoramento
3. Executar teste conforme cenário
4. Coletar métricas
5. Encerrar execução
6. Consolidar dados

Incluir:

* Ferramenta de carga (ex: k6)
* Monitoramento (ex: Grafana + Prometheus)
* Ambiente (homologação, staging, produção controlada)

---

## 7️⃣ Critérios de Sucesso e Falha

Definir critérios objetivos.

### Critérios de Sucesso:

* p95 ≤ 500ms
* Taxa de erro ≤ 1%
* CPU ≤ 75%
* Sistema estável durante todo o teste

### Critérios de Falha:

* Tempo de resposta acima do limite definido
* Taxa de erro superior ao aceitável
* Queda de serviço
* Saturação de recursos

Sem critério definido, não há decisão técnica.

---

## 8️⃣ Recursos Necessários

### Humanos:

* QA de Performance
* Dev Backend
* DevOps / Infraestrutura
* DBA (se necessário)

### Técnicos:

* Ferramenta de carga
* Ambiente provisionado
* Monitoramento ativo
* Massa de dados
* Acesso aos logs

---

## 9️⃣ Relatório de Teste

Documento final contendo:

* Resumo executivo
* Objetivo do teste
* Cenário executado
* Métricas coletadas
* Gráficos (tempo de resposta, RPS, CPU)
* Análise técnica
* Gargalos identificados
* Recomendações
* Decisão (Aprovado / Reprovado)

⚠ O relatório deve ser claro tanto para técnico quanto para gestor.

---

# 🧠 Observação Estratégica

Um Plano de Teste de Performance maduro:

✔ Está alinhado com risco de negócio
✔ Define meta antes de executar
✔ Relaciona aplicação + infraestrutura
✔ Gera decisão clara
