# 📌 O que são SLI (Service Level Indicators)?

SLI são **métricas mensuráveis que indicam a saúde real do sistema**.

Eles respondem:

> O serviço está funcionando dentro do esperado?

SLI não é meta.
SLI é medição.

Depois vem:

* SLO → objetivo interno
* SLA → acordo contratual

Hoje vamos focar só no que medir.

---

# 🧱 Categorias de Métricas em Performance

Dividimos em 3 grandes blocos:

1️⃣ Métricas de Aplicação
2️⃣ Métricas de Infraestrutura
3️⃣ Métricas de Experiência do Usuário

---

# 1️⃣ Métricas de Aplicação

Essas são as principais para QA de Performance.

---

## 🔹 Tempo de Resposta (Response Time)

Tempo total para uma requisição ser processada.

Sempre analisar:

* Média (mean)
* Mediana
* p90
* p95
* p99

⚠ Nunca confiar só na média.

---

## 🔹 Percentis (p95 / p99)

Mostram a experiência da maioria dos usuários.

Exemplo:

* p95 = 450ms
  → 95% das requisições responderam abaixo de 450ms

Se p99 for muito maior que p95:
→ Existe cauda longa (requisições lentas específicas)

Isso pode indicar:

* Query lenta
* Lock de banco
* Garbage collection
* Contenção de recurso

---

## 🔹 Throughput (RPS – Requests per Second)

Quantas requisições o sistema processa por segundo.

Exemplo:

* 200 RPS

Importante:
Se aumentar usuários e RPS não subir → sistema saturou.

---

## 🔹 Taxa de Erro (%)

Percentual de requisições com falha.

Pode incluir:

* HTTP 5xx
* Timeout
* Falha de conexão
* Erro de aplicação

Exemplo:

* 2% erro → crítico em produção.

---

## 🔹 Apdex (Application Performance Index)

Índice de satisfação do usuário.

Divide respostas em:

* Satisfeito
* Tolerável
* Frustrado

Escala de 0 a 1.

Quanto mais perto de 1, melhor.

---

# 2️⃣ Métricas de Infraestrutura

Aqui você encontra gargalos.

---

## 🔹 Uso de CPU

Se CPU fica constantemente acima de 80–90%:
→ Sistema pode estar saturando.

Mas cuidado:
CPU alta não é sempre problema.
CPU baixa com sistema lento → gargalo pode ser I/O ou banco.

---

## 🔹 Uso de Memória

Monitorar:

* Crescimento contínuo
* Swap
* Memory leak

Se memória sobe progressivamente em teste longo:
→ possível vazamento.

---

## 🔹 I/O de Disco

Importante para:

* Sistemas com logs intensos
* Banco de dados
* Processamento de arquivos

Disco lento pode travar tudo.

---

## 🔹 Conexões de Banco

Verificar:

* Pool de conexões
* Locks
* Deadlocks
* Query time

Muitos problemas de performance estão no banco.

---

## 🔹 Latência de Rede

Importante em:

* Microserviços
* Sistemas distribuídos
* Cloud multi-região

Alta latência pode parecer lentidão da aplicação.

---

# 3️⃣ Métricas de Experiência do Usuário (Frontend)

Para páginas web:

---

## 🔹 TTFB (Time To First Byte)

Tempo até o primeiro byte chegar.

Indica:

* Latência
* Processamento inicial

---

## 🔹 Tempo de Carregamento Total

Tempo até página estar totalmente renderizada.

---

## 🔹 Core Web Vitals

Incluem:

* LCP (Largest Contentful Paint)
* CLS (Cumulative Layout Shift)
* INP/FID (Interatividade)

Essas métricas impactam SEO e experiência.

---

# 📊 Como escolher bons SLI?

Você deve perguntar:

* O que impacta o negócio?
* O que impacta o usuário?
* Onde o sistema costuma falhar?

Para API crítica:
→ Tempo de resposta + taxa de erro

Para e-commerce:
→ p95 + checkout + taxa de erro

Para sistema bancário:
→ latência + disponibilidade + consistência

---

# 🚨 Erros Comuns com Métricas

❌ Usar só média
❌ Ignorar infraestrutura
❌ Não correlacionar erro com CPU
❌ Não definir meta antes do teste
❌ Medir tudo e não analisar nada

---

# 🧠 Mentalidade Sênior

Não é só coletar métrica.

É cruzar informações.

Exemplo:

* RPS sobe
* CPU sobe
* Tempo de resposta sobe
* Taxa de erro sobe

Conclusão provável:
→ Saturação de processamento

Outro exemplo:

* CPU normal
* Memória normal
* Tempo sobe
* Conexões de banco no limite

Conclusão:
→ Gargalo no banco

---

# 🎯 Resumo Estratégico

SLI responde:

✔ O sistema está saudável?
✔ O usuário está satisfeito?
✔ A infraestrutura está saturando?
✔ O negócio está protegido?
